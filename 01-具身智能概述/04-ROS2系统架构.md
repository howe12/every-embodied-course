## 1. ROS2分层架构——像搭积木一样理解系统

### 1.1 三层架构总览

想象一下你在使用手机。手机有硬件（屏幕、芯片）、有操作系统（Android/iOS）、还有各种APP应用。ROS2的系统架构与此类似，也是分层的。

![ROS2分层架构图](./images/ros2_layered_architecture.jpg)
*图注：ROS2 系统分为三层：OS层、中间件层、应用层*

**ROS2分层架构自下而上分为三层：**

| 层级 | 名称 | 说明 | 例子 |
|------|------|------|------|
| 底层 | OS层 | 操作系统抽象层 | Linux、Windows、macOS |
| 中间层 | 中间件层 | 通讯核心（DDS） | Fast DDS、Cyclone DDS、Connext DDS |
| 上层 | 应用层 | 功能包和节点 | navigation2、slam_toolbox |

这种分层设计的好处是：**解耦**。上层不需要关心底层用什么操作系统，DDS厂商也不需要了解你的机器人具体在做什么应用。每一层都可以独立演进，互不干扰。

### 1.2 OS层——跨平台的基石

ROS2能运行在Linux、Windows、macOS上，靠的就是这层抽象。无论你用什么操作系统，ROS2都提供统一的API。

ROS2支持的操作系统：
- **Linux**：Ubuntu 20.04/22.04、Debian、RHEL
- **Windows**：Windows 10/11
- **macOS**：macOS 10.14+

这意味着，你可以在Windows上开发，在Linux上部署，或者在嵌入式设备上运行——只要安装对应的ROS2版本即可。

### 1.3 中间件层——ROS2的心脏

中间件层是ROS2的核心，它基于**DDS（Data Distribution Service）**实现。

DDS负责：
- 节点发现（谁在运行？）
- 数据传输（发什么数据？）
- 服务质量（要不要保证数据到达？）

你不需要直接操作DDS，ROS2已经封装好了。但理解DDS的工作原理，能帮你更好地调优通讯性能。

### 1.4 应用层——你的代码

最上层是你编写的机器人代码——也就是ROS2中的**节点（Node）**。

每个节点是一个独立的程序，负责特定功能：
- `camera_node` 负责读取摄像头
- `laser_scan_node` 负责发布激光雷达数据
- `navigation_node` 负责路径规划

节点之间通过话题、服务、动作进行通讯，组成了完整的机器人软件系统。

## 2. 核心组件——ROS2的四大天王

### 2.1 节点（Node）——功能模块

**节点**是ROS2中最基本的执行单元。每个节点是一个独立的可执行程序，负责单一功能。

![ROS2节点示意图](./images/ros2_node_example.jpg)
*图注：每个节点是一个独立程序，负责特定功能*

**节点的特点：**
- **单一职责**：一个节点只做一件事
- **可执行**：节点是一个运行中的程序
- **独立进程**：通常每个节点运行在独立进程中

**查看节点列表：**
```bash
# 列出当前运行的节点
ros2 node list
```

**查看节点信息：**
```bash
# 查看特定节点的详细信息
ros2 node info /node_name
```

### 2.2 话题（Topic）——广播通讯

**话题**是ROS2中最常用的通讯方式，采用**发布-订阅**模式。

![ROS2话题通讯示意图](./images/ros2_topic_communication.jpg)
*图注：发布者发布数据，订阅者接收数据，无需直接连接*

**话题的特点：**
- **异步通讯**：发布者不需要等待订阅者
- **一对多**：一个发布者可以对应多个订阅者
- **多对多**：多个发布者可以发布同一话题

**常见话题示例：**
| 话题名 | 数据类型 | 用途 |
|--------|----------|------|
| `/scan` | LaserScan | 激光雷达数据 |
| `/cmd_vel` | Twist | 运动控制指令 |
| `/image` | Image | 摄像头图像 |

**常用命令：**
```bash
# 列出所有话题
ros2 topic list

# 查看话题消息类型
ros2 topic type /scan

# 实时显示话题数据
ros2 topic echo /scan
```

### 2.3 服务（Service）——请求-响应

**服务**是ROS2中的同步通讯方式，采用**请求-响应**模式，就像调用函数一样。

![ROS2服务通讯示意图](./images/ros2_service.jpg)
*图注：客户端发送请求，服务端处理后返回响应*

**服务的特点：**
- **同步通讯**：客户端等待服务端响应
- **一对多**：一个服务只能有一个服务端，但可有多个客户端
- **短任务**：适合查询、控制等短时操作

**常见服务示例：**
| 服务名 | 类型 | 用途 |
|--------|------|------|
| `/spawn` | Spawn | 生成新实体 |
| `/clear` | Empty | 清除地图 |

**常用命令：**
```bash
# 列出所有服务
ros2 service list

# 调用服务
ros2 service call /spawn turtlesim/srv/Spawn "{x: 3, y: 3, theta: 0}"
```

### 2.4 动作（Action）——带反馈的任务

**动作**是ROS2中用于长时间任务的高级通讯机制，支持**目标、反馈、结果**三部分。

![ROS2动作通讯示意图](./images/ros2_action.jpg)
*图注：动作支持目标执行、过程反馈、完成结果*

**动作的特点：**
- **可取消**：可以在执行过程中取消目标
- **带反馈**：实时反馈任务进度
- **可结果**：任务完成后返回结果
- **长时间**：适合导航、机械臂运动等耗时的任务

**常见动作示例：**
| 动作名 | 类型 | 用途 |
|--------|------|------|
| `/move_base` | Nav2Action | 导航到目标点 |
| `/follow_joint_trajectory` | FollowJointTrajectory | 机械臂轨迹跟踪 |

### 2.5 参数（Parameter）——配置管理

**参数**用于配置节点的行为，就像程序的配置文件一样。

**参数的特点：**
- **运行时可修改**：无需重启节点
- **类型丰富**：支持整型、浮点型、字符串、布尔值等
- **层级管理**：按节点名称组织

**常用命令：**
```bash
# 列出节点参数
ros2 param list /node_name

# 获取参数值
ros2 param get /node_name param_name

# 设置参数值
ros2 param set /node_name param_name value
```

### 2.6 坐标系（TF）——机器人位置关系

**TF（Transform）**是ROS2中管理坐标系变换的系统，让机器人知道"自己在哪"和"物体在哪"。

![ROS2 TF坐标系示意](./images/ros2_tf_tree.jpg)
*图注：TF树管理各坐标系之间的变换关系*

**常见坐标系：**
| 坐标系 | 说明 |
|--------|------|
| `map` | 全局地图坐标系 |
| `odom` | 里程计坐标系 |
| `base_link` | 机器人基座坐标系 |
| `base_scan` | 传感器坐标系 |

**常用命令：**
```bash
# 查看TF树
ros2 run tf2_ros view_frames

# 监听特定坐标变换
ros2 run tf2_ros tf2_echo source_frame target_frame
```

## 3. DDS数据分发原理——深入通讯核心

### 3.1 什么是DDS？

**DDS（Data Distribution Service）**是对象管理组织（OMG）制定的一种分布式实时通讯标准。

你可以把DDS想象成**物联网时代的"路由器"**——它负责在各个节点之间路由数据，确保数据能正确、及时地送达。

![DDS架构示意图](./images/dds_architecture.jpg)
*图注：DDS 采用全局数据空间（Global Data Space）概念*

**DDS的核心概念：**

- **全域（Domain）**：一个DDS域是一个独立的通讯空间，不同域之间隔离
- **参与者（Participant）**：参与通讯的实体
- **发布者（Publisher）**：负责发送数据
- **订阅者（Subscriber）**：负责接收数据
- **主题（Topic）**：数据的"类型"，类似ROS2话题

### 3.2 发布-订阅模式

DDS采用**发布-订阅（Publish-Subscribe）**模式，这与ROS2的话题通讯完全一致。

![DDS发布订阅模式](./images/dds_pubsub.jpg)
*图注：发布者将数据写入全局数据空间，订阅者从空间读取*

**工作流程：**
1. 发布者创建数据，写入"全局数据空间"
2. DDS自动将数据分发给订阅了对应主题的订阅者
3. 订阅者从"全局数据空间"读取数据

这种模式的优点是：**完全解耦**。发布者和订阅者不需要知道彼此的存在，甚至可以跨机器、跨网络通讯。

### 3.3 QoS服务质量配置

**QoS（Quality of Service）**是DDS最强大的特性之一，它允许你配置通讯的"服务质量"。

![ROS2 QoS配置示意](./images/ros2_qos.jpg)
*图注：ROS2 支持多种 QoS 策略组合*

**常见的QoS策略：**

| QoS策略 | 说明 | 适用场景 |
|---------|------|----------|
| `RELIABLE` | 保证数据到达 | 控制指令 |
| `BEST_EFFORT` | 尽快到达，不保证 | 传感器数据 |
| `KEEP_LAST(n)` | 保留最近n条 | 历史数据查询 |
| `TRANSIENT_LOCAL` | 历史数据持久化 | 新节点加入 |

**ROS2中的QoS配置：**
```python
from rclpy.qos import QoSProfile, ReliabilityPolicy, HistoryPolicy

# 创建可靠通讯的QoS配置
qos = QoSProfile(
    reliability=ReliabilityPolicy.RELIABLE,
    history=HistoryPolicy.KEEP_LAST,
    depth=10
)
```

### 3.4 节点发现机制

DDS的**自动发现机制**是其核心优势之一。

![DDS节点发现流程](./images/dds_discovery.jpg)
*图注：DDS 通过简单参与者协议（SPDP）自动发现节点*

**发现过程：**
1. 节点启动时，向域内广播"我在这里"
2. 其他节点收到广播后，建立连接
3. 节点之间直接通讯，无需中间服务器

这种机制的好处是：**去中心化**。没有Master节点，任意节点崩溃都不会影响其他节点通讯。

### 4.3 实训步骤

#### 步骤1：创建发布者节点

**目标**：创建一个发布"Hello World"消息的节点

**代码实现**：
```python
# 创建文件 ~/ros2_ws/src/my_first_pkg/my_first_pkg/publisher_node.py

import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class PublisherNode(Node):
    def __init__(self):
        super().__init__('publisher_node')
        self.publisher_ = self.create_publisher(String, 'chatter', 10)
        self.timer = self.create_timer(1.0, self.timer_callback)
        self.count = 0

    def timer_callback(self):
        msg = String()
        msg.data = f'Hello World {self.count}'
        self.publisher_.publish(msg)
        self.get_logger().info(f'Publishing: "{msg.data}"')
        self.count += 1

def main(args=None):
    rclpy.init(args=args)
    node = PublisherNode()
    rclpy.spin(node)
    node.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()
```

**运行**：
```bash
# 编译包
cd ~/ros2_ws
colcon build --packages-select my_first_pkg

# 启动发布者
ros2 run my_first_pkg publisher_node
```

**验证**：
```bash
# 查看话题列表
ros2 topic list

# 监听话题数据
ros2 topic echo /chatter
```

#### 步骤2：创建订阅者节点

**目标**：创建一个订阅"chatter"话题的节点

**代码实现**：
```python
# 创建文件 ~/ros2_ws/src/my_first_pkg/my_first_pkg/subscriber_node.py

import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class SubscriberNode(Node):
    def __init__(self):
        super().__init__('subscriber_node')
        self.subscription = self.create_subscription(
            String,
            'chatter',
            self.listener_callback,
            10
        )

    def listener_callback(self, msg):
        self.get_logger().info(f'I heard: "{msg.data}"')

def main(args=None):
    rclpy.init(args=args)
    node = SubscriberNode()
    rclpy.spin(node)
    node.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()
```

**运行**：
```bash
# 重新编译
colcon build --packages-select my_first_pkg

# 启动订阅者（先启动发布者，再启动订阅者）
ros2 run my_first_pkg subscriber_node
```

#### 步骤3：创建服务服务器

**目标**：创建一个返回自定义消息的服务

**代码实现**：
```python
# 创建文件 ~/ros2_ws/src/my_first_pkg/my_first_pkg/service_server.py

import rclpy
from rclpy.node import Node
from example_interfaces.srv import AddTwoInts

class ServiceServer(Node):
    def __init__(self):
        super().__init__('service_server')
        self.service = self.create_service(
            AddTwoInts,
            'add_two_ints',
            self.add_two_ints_callback
        )

    def add_two_ints_callback(self, request, response):
        response.sum = request.a + request.b
        self.get_logger().info(f'Incoming request: a={request.a}, b={request.b}, sum={response.sum}')
        return response

def main(args=None):
    rclpy.init(args=args)
    node = ServiceServer()
    rclpy.spin(node)
    node.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()
```

**调用服务**：
```bash
# 启动服务服务器
ros2 run my_first_pkg service_server

# 另一个终端调用服务
ros2 service call /add_two_ints example_interfaces/srv/AddTwoInts "{a: 5, b: 3}"
```

## 本章小结

本章我们深入学习了ROS2的系统架构和核心概念。

**核心要点回顾：**

1. **分层架构**：ROS2分为OS层、中间件层、应用层，三层解耦、互不干扰
2. **核心组件**：
   - 节点（Node）是基本执行单元
   - 话题（Topic）采用发布-订阅模式，适合异步通讯
   - 服务（Service）采用请求-响应模式，适合短时操作
   - 动作（Action）支持反馈和取消，适合长时间任务
   - 参数（Parameter）用于运行时配置
   - TF用于管理坐标系变换
3. **DDS通讯原理**：基于发布-订阅模式，支持QoS配置和自动节点发现
