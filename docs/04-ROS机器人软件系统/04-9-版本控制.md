## 1. Git基础操作

### 1.1 初始化和克隆

Git是目前最流行的分布式版本控制系统。在ROS2项目中使用Git，需要了解基本的初始化和克隆操作。

```bash
# 初始化新仓库
cd ~/ros2_ws/src
git init

# 克隆已有仓库
git clone https://github.com/ros2/examples.git

# 克隆指定分支
git clone -b humble https://github.com/ros2/examples.git
```

![Git工作区结构](./images/git_areas.jpg)
*图注：Git的三个工作区域*

理解Git的三个区域非常重要：工作目录（Working Directory）是你编辑文件的地方；暂存区（Staging Area）保存了即将提交的文件快照；版本库（Repository）保存了所有历史提交记录。

### 1.2 基本工作流程

```bash
# 查看当前状态
git status

# 查看文件变更
git diff

# 添加文件到暂存区
git add src/my_package/
git add .

# 提交变更
git commit -m "Add my_package: initial implementation"

# 查看提交历史
git log --oneline -10

# 查看远程仓库
git remote -v
```

对于ROS2项目，建议按功能包进行提交。这样每个提交都是独立的功能单元，便于追踪和管理。

### 1.3 同步远程仓库

```bash
# 获取远程更新
git fetch origin

# 拉取并合并
git pull origin main

# 推送本地提交
git push origin main
```

在团队协作中，保持本地仓库与远程同步非常重要。建议在开始工作前和提交代码前都执行一次pull操作，避免合并冲突。

## 2. 分支管理策略

### 2.1 常用分支类型

规范的分支管理是团队协作的基础。推荐使用以下分支策略：

| 分支类型 | 用途 | 命名示例 |
|----------|------|----------|
| main/master | 主分支，保持稳定可发布 | - |
| develop | 开发分支，集成新功能 | - |
| feature/* | 功能开发分支 | feature/navigation |
| bugfix/* | Bug修复分支 | bugfix/laser_driver |
| release/* | 发布准备分支 | release/v1.0 |
| hotfix/* | 紧急修复分支 | hotfix/safety_issue |

![Git分支策略](./images/git_branching.jpg)
*图注：Git分支工作流*

### 2.2 功能分支开发流程

```bash
# 1. 从develop创建功能分支
git checkout -b feature/my_new_feature develop

# 2. 开发完成后，切换回develop
git checkout develop

# 3. 合并功能分支
git merge --no-ff feature/my_new_feature

# 4. 删除功能分支
git branch -d feature/my_new_feature
```

使用--no-ff参数可以保留功能分支的完整历史，便于追踪每个功能的开发过程。

### 2.3 处理合并冲突

当多个分支修改了同一文件的同一部分时，会发生合并冲突。

```bash
# 1. 尝试合并
git merge feature/my_feature

# 2. 查看冲突文件
git status

# 3. 手动解决冲突
# 编辑冲突文件，保留需要的代码
# <<<<<<< HEAD
# 当前分支的代码
# =======
# 要合并的分支的代码
# >>>>>>> feature/my_feature

# 4. 标记冲突已解决
git add .

# 5. 完成合并提交
git commit -m "Merge feature/my_feature into develop"
```

解决冲突时，需要仔细理解两边的代码逻辑，选择或合并正确的代码。ROS2项目中，常见的冲突发生在package.xml和CMakeLists.txt的依赖声明部分。

## 3. Pull Request流程

### 3.1 创建Pull Request

Pull Request（PR）是代码审核和协作的核心机制。在ROS2项目中，推荐使用GitHub或GitLab托管代码。

创建PR的流程：

1. Fork项目仓库（如果你是贡献者）
2. 克隆仓库到本地
3. 创建功能分支并开发
4. 推送分支到远程
5. 在Web界面上创建PR

![Pull Request流程](./images/pull_request.jpg)
*图注：Pull Request工作流程*

### 3.2 代码审核要点

收到PR审核请求时，应重点检查以下方面：

**代码质量**

- 代码逻辑是否正确
- 是否符合项目的代码风格
- 是否有潜在的Bug或错误
- 错误处理是否完善

**ROS2规范**

- package.xml依赖是否正确声明
- CMakeLists.txt配置是否规范
- 节点命名是否遵循ROS2规范
- 话题/服务/动作命名是否清晰

**文档和测试**

- 是否添加了必要的注释
- 是否有单元测试或集成测试
- API变更是否有说明

### 3.3 评论和修改

PR审核过程中，审核者会提出修改意见。开发者需要根据意见修改代码并更新PR。

```bash
# 1. 在原分支上继续修改
git checkout feature/my_feature

# 2. 修改代码
vim src/my_package/src/node.cpp

# 3. 提交修改
git add src/my_package/src/node.cpp
git commit -m "Fix: address review comments"

# 4. 推送到远程
git push origin feature/my_feature
```

PR会自动更新，审核者可以看到新的提交。

## 4. 代码规范检查

### 4.1 clang-format代码格式化

clang-format是C++代码格式化工具，可以自动统一代码风格。

```bash
# 安装clang-format
sudo apt install clang-format

# 格式化指定文件
clang-format -i src/my_package/src/node.cpp

# 格式化整个包
find my_package -name "*.cpp" -o -name "*.hpp" | xargs clang-format -i
```

ROS2项目通常提供.clang-format配置文件，放置在功能包根目录：

```yaml
# .clang-format 配置文件示例
BasedOnStyle: google
IndentWidth: 4
ColumnLimit: 100
```

### 4.2 ament_lint代码检查

ROS2提供了ament_lint工具集，用于自动检查ROS2包的规范。

```bash
# 运行lint检查
colcon test --packages-select my_package

# 单独运行检查
ament_lint_cmake my_package
ament_lint_auto my_package
```

常见的lint检查包括：文件权限检查、版权声明检查、CMake规范检查、C++编码规范检查等。

### 4.3 在CI/CD中集成

自动化检查可以集成到持续集成流程中：

```yaml
# .github/workflows/ci.yml 示例
name: CI

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - uses: ros-tooling/setup-ros@0.6
      - name: Build
        run: colcon build
      - name: Test
        run: colcon test
      - name: Lint
        run: |
          source install/setup.bash
          ament_lint_auto
```
