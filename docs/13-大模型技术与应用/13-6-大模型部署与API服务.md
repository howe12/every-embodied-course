## 1. 模型部署概述

### 1.1 本地部署 vs 云端部署

将训练好的大模型投入实际应用，首先需要选择合适的部署方式。根据计算资源的位置和调用方式的不同，部署方案可分为**本地部署**和**云端部署**两大类。

#### 本地部署（On-Premise）

本地部署是指将模型运行在本地服务器或边缘设备上，所有计算均在本地完成。

| 优势 | 劣势 |
|------|------|
| **数据隐私**：敏感数据无需离开本地，满足 GDPR 等合规要求 | **硬件成本高**：需要采购和维护 GPU 服务器 |
| **低延迟**：无需网络通信，本地推理延迟更低 | **扩展性差**：增加并发需要增购硬件 |
| **离线可用**：不依赖网络连接，适合机器人等嵌入式场景 | **资源利用率低**：模型难以被多个应用共享 |
| **可控性强**：可深度定制优化策略 | **运维负担重**：需要专业团队维护 |

**适用场景**：机器人实时推理、医疗数据处理、金融合规场景、隐私敏感业务。

#### 云端部署（Cloud Deployment）

云端部署将模型托管在云服务商（如 AWS、阿里云、Azure）的 GPU 集群上，通过 API 调用。

| 优势 | 劣势 |
|------|------|
| **弹性扩展**：根据请求量自动扩缩容，按需付费 | **数据隐私风险**：数据需传输到第三方服务器 |
| **免运维**：云厂商负责硬件维护和故障恢复 | **网络延迟**：受网络状况影响 |
| **全球可用**：通过 API 随时随地访问 | **成本可能较高**：高频调用场景成本显著 |
| **最新模型**：云端可快速更新到最新模型版本 | **供应商锁定**：迁移成本较高 |

**适用场景**：通用对话服务、API 即服务平台、流量波动的业务场景。

#### 混合部署架构

实际应用中，常见的是**混合部署**：本地部署用于低延迟、实时推理（机器人端），云端用于复杂推理和离线批处理（服务端）。

### 1.2 推理优化目标

大模型推理与训练有本质区别，理解推理阶段的核心挑战是优化的前提。

#### 训练 vs 推理

| 维度 | 训练（Training） | 推理（Inference） |
|------|-----------------|------------------|
| **模式** | 自回归生成，需要全部 Token 的梯度 | 同样自回归生成，但无需梯度 |
| **计算模式** | 数据并行，大 Batch | 单请求或小 Batch |
| **延迟要求** | 不严格，数小时到数天均可 | 严格，尤其是交互式场景 |
| **内存占用** | 模型权重 + 优化器 + 梯度 + 激活值 | 主要为模型权重 + KV Cache |
| **批处理** | 高效，大 Batch 利用率高 | 低效，请求随机到达 |

#### 核心优化指标

**1. 吞吐量（Throughput）**

吞吐量指单位时间内系统处理的总 Token 数，是衡量系统处理能力的核心指标。

$$
\text{Throughput} = \frac{\text{总输出 Token 数}}{\text{总时间}} \quad \text{(tokens/s)}
$$

在批处理场景中，吞吐量是最重要的优化目标。

**2. 首 Token 延迟（Time to First Token, TTFT）**

首 Token 延迟指从发送请求到收到第一个输出 Token 的时间，直接影响用户的交互体验。

$$
\text{TTFT} = \underbrace{\frac{\text{Prefill Token 数}}{\text{ Prefill 速度}}}_{\text{计算延迟}} + \underbrace{\text{排队等待时间}}_{\text{调度延迟}}
$$

Prefill 阶段需要处理整个输入 Prompt，是 KV Cache 填充阶段。

**3. 单 Token 延迟（Time Per Output Token, TPOT）**

单 Token 延迟指生成每个输出 Token 所需的平均时间：

$$
\text{TPOT} = \frac{\text{单次推理总时间} - \text{Prefill 时间}}{\text{输出 Token 数} - 1}
$$

TPOT 反映了推理系统在 Decode 阶段的速度。

**4. 显存占用（VRAM Usage）**

大模型推理的显存占用由以下部分组成：

| 组件 | 说明 | 典型占比（7B 模型 FP16） |
|------|------|--------------------------|
| **模型权重** | $\text{参数量} \times 2 \text{ bytes}$ | ~14GB（7B FP16） |
| **KV Cache** | $2 \times \text{layers} \times \text{batch} \times \text{seq\_len} \times \text{head\_dim} \times 2$ | 动态，与序列长度成正比 |
| **激活值** | 中间计算结果 | 与 Batch 大小相关 |
| **注意力矩阵** | Softmax 注意力计算 | 可优化 |

对于 7B 参数模型（FP16）：
- 纯权重：$7 \times 10^9 \times 2 \text{ bytes} \approx 14\text{GB}$
- 加上 KV Cache 和激活值，单卡（24GB）勉强能运行，但并发受限

**5. 内存带宽利用率（Memory Bandwidth Utilization, MBU）**

推理性能很大程度上受限于显存带宽。MBU 反映了系统对显存带宽的利用程度：

$$
\text{MBU} = \frac{\text{实际显存带宽}}{\text{峰值显存带宽}}
$$

典型 GPU 峰值带宽：NVIDIA A100 40GB 为 2TB/s，H100 80GB 为 3.35TB/s。

### 1.3 量化技术

**量化（Quantization）** 是将模型参数从高精度（FP32/FP16）转换为低精度（INT8/INT4/INT2）表示的技术，是降低显存占用和加速推理最有效的手段之一。

#### 量化基本原理

原始浮点数 $x_f$ 映射到量化值 $x_q$：

$$
x_q = \text{round}\left(\frac{x_f}{s}\right) + z
$$

其中 $s$ 为缩放因子（scale），$z$ 为零点（zero point）。

反量化：

$$
\hat{x}_f = s \cdot (x_q - z)
$$

#### 量化精度等级

| 格式 | 位宽 | 显存减少 | 精度损失 | 适用场景 |
|------|------|----------|----------|----------|
| **FP16** | 16-bit | 1× | 无 | 基线 |
| **BF16** | 16-bit | 1× | 极小 | 精度敏感场景，A100/H100 |
| **INT8** | 8-bit | 2× | 较小 | 通用场景，主流选择 |
| **INT4** | 4-bit | 4× | 中等 | 边端部署，内存受限 |
| **INT2** | 2-bit | 8× | 较大 | 实验性研究 |
| **NF4** | 4-bit | 4× | 极小 | GPTQ/AWQ 配套格式 |

#### 量化方案对比

| 方案 | 类型 | 精度 | 速度 | 显存 | 代表工具 |
|------|------|------|------|------|---------|
| **FP16** | 原始 | 基准 | 基准 | 基准 | - |
| **动态 INT8** | 动态 | ~原始 | 1.2-1.5× | ~50% | PyTorch `torch.nn.functional.quantize_dynamic` |
| **静态 INT8** | 静态 | ~原始 | 1.5-2× | ~50% | TensorRT、ONNX Runtime |
| **GPTQ** | PTQ | ~95% | 2-4× | ~25% | `auto-gptq`、`gptq` |
| **AWQ** | PTQ | ~97% | 2-4× | ~25% | `awq`、`llm-awq` |
| **GGUF（Q4_K/M）** | QLoRA 配套 | ~96% | 2-3× | ~25% | llama.cpp |
| **bitsandbytes INT8** | PTQ | ~95% | 1.2-1.5× | ~50% | `bitsandbytes` |

#### GPTQ 量化

**GPTQ（Generative Post-Training Quantization）** 是最流行的 INT4 量化方法之一，在保持模型性能的同时大幅降低显存需求。

**核心思想**：对角线重构（Per-channel quantization），逐层量化并补偿重构误差。

**数学推导**：

给定权重矩阵 $W$，量化目标为找到 $\hat{W}$ 使重构误差最小：

$$
\hat{W} = \arg\min_{\hat{W}} \|W - \hat{W}\|_F^2
$$

GPTQ 通过近似求解 Hessian 矩阵的逆来计算最优量化权重。

```python
# GPTQ 量化示例（使用 auto-gptq）
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig
from transformers import AutoTokenizer

# 模型路径和输出路径
model_name = "./models/llama-7b"
output_dir = "./models/llama-7b-gptq-4bit"

# 加载模型
print("加载模型中...")
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoGPTQForCausalLM.from_pretrained(
    model_name,
    quantize_config=BaseQuantizeConfig(
        bits=4,           # 量化位宽（4-bit）
        group_size=128,   # 每组参数量，越小精度越高但速度越慢
        desc_act=True,   # 激活顺序重排，对质量有帮助但稍慢
    )
)

# 准备校准数据（用于校准量化参数）
# 通常使用 128-1024 条代表性样本
calibration_data = [
    tokenizer("样本文本 " + str(i), return_tensors="pt")
    for i in range(512)
]

# 执行量化
print("开始量化（此过程需要数分钟）...")
model.quantize(calibration_data)

# 保存量化模型
model.save_quantized(output_dir)
print(f"量化完成，模型已保存至: {output_dir}")

# 量化后模型加载
# quantized_model = AutoGPTQForCausalLM.from_quantized(
#     output_dir,
#     device_map="auto"
# )
```

#### AWQ 量化

**AWQ（Activation-Aware Weight Quantization）** 是 2024 年提出的更先进量化方法，相比 GPTQ 精度更高（~97% 保留）。

**核心洞察**：并非所有权重都同等重要——与大幅激活值相乘的权重对精度贡献更大，应使用更高精度。

**AWQ 量化步骤**：

1. 计算各层激活幅值（activation magnitude）
2. 搜索最优缩放因子（scale factor）
3. 仅量化权重，激活值保持 FP16
4. 执行 INT4 量化

```python
# AWQ 量化示例（使用 llm-awq）
# pip install awq

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "./models/llama-7b"
quant_path = "./models/llama-7b-awq-4bit"

# 加载模型
model = AutoAWQForCausalLM.from_pretrained(model_path)
tokenizer = AutoTokenizer.from_pretrained(model_path)

# 定义量化配置
quant_config = {
    "zero_point": True,       # 使用非对称量化
    "q_group_size": 128,      # 量化组大小
    "w_bit": 4,               # 权重位宽
    "version": "GEMM",         # GEMM 量化版本（速度优先）
}

# 校准数据
def get_calib_data():
    data = []
    for i in range(128):
        text = f"校准样本 {i}：" + "这是一段用于 AWQ 量化校准的代表性文本。"
        data.append(tokenizer(text, return_tensors="pt"))
    return data

print("开始 AWQ 量化...")
model.quantize(
    get_calib_data(),
    quant_config=quant_config
)

# 保存
model.save_quantized(quant_path)
print(f"AWQ 量化完成: {quant_path}")
```

#### GGUF 格式（llama.cpp）

**GGUF（GPT-Generated Unified Format）** 是 llama.cpp 推出的模型格式，专门针对 CPU 和混合推理优化，是本地部署最流行的格式之一。

**GGUF 的优势**：

| 特性 | 说明 |

| **内存映射（mmap）** | 大模型文件映射到内存，无需全部加载，按需读取 |
| **多文件分片** | 支持将大模型分片存储在多个小文件中 |

| **快速推理** | SIMD 优化（AVX2/AVX512/NEON），CPU 推理效率高 |

## 2. 推理框架详解

### 2.1 vLLM

**vLLM** 是目前最流行的开源大模型推理框架，由加州大学伯克利分校开发，核心创新是 **PagedAttention** 算法，在保持模型质量的同时将推理吞吐量提升至 HF（Hugging Face）的 **24 倍**以上。

#### PagedAttention

传统推理中，KV Cache 按键值对存储为连续 Tensor。随着对话序列增长，已生成的 Token 的 KV Cache 需要不断追加存储，但预分配的连续内存可能耗尽，导致需要将已生成序列的 KV Cache 从 GPU 显存换出到 CPU 内存，极大降低吞吐量。

**PagedAttention 的核心思想**：将 KV Cache 的存储管理类比为操作系统的虚拟内存分页管理。

```
传统方式（连续存储）：

Token:    [ 系统 ] [ 你好 ] [ 请问 ] [ 今天 ] [ 天气 ]
KV Cache: [  12  ] [  45  ] [  67  ] [  23  ] [  89  ]  ← 连续内存，必须预分配
           ↑                                        ↑
        固定起点                                长度受限

PagedAttention（分页存储）：

Token:    [ 系统 ] [ 你好 ] [ 请问 ] [ 今天 ] [ 天气 ]
KV Cache: [  12  ] [  45  ] [  67  ] [  23  ] [  89  ]
  Page 0   ──────  ──────  ──────                   ↑
  Page 1                                          按需分配
  Page 2   ←─────── 动态扩展，可共享，可换出
```

**关键特性**：
- KV Cache 按 Block 分页（通常 16 个 Token/Block）
- 不同序列的 KV Cache 可以共享物理页面（用于 Beam Search 和前缀复用）
- 显存按需分配，消除因内存碎片导致的浪费

**PagedAttention 内存效率**：

$$
\text{显存节省} = \frac{\text{传统预分配显存}}{\text{PagedAttention 实际使用}} \approx 50\% - 80\%
$$

这使得 vLLM 能在相同硬件上支持更多并发请求。

#### Continuous Batching（持续批处理）

**Continuous Batching** 是 vLLM 的另一核心优化。相比传统的 Static Batching（等待一批请求全部完成后才开始下一批），Continuous Batching 允许新请求在任何时刻加入正在执行的批次。

```
Static Batching（静态批处理）：

时间 →████████████████████████████████████████████

批次1:  [请求A: 生成长回复] [请求B: 生成长回复] [请求C: 生成长回复]
        ←───── 都完成后才能开始下一批 ─────→
        ←─ B和C 早已完成，却要等 A ─→  资源浪费

Continuous Batching（持续批处理）：

时间 →████████████████████████████████████████████

批次1:  [A●●●●●●●●●●] [B●] [C●●●●●]    ← B 和 C 完成后立即插入新请求
批次2:            [D●●●●●●●●●●●●] [E●●●] [F●●●●]
批次3:                              [G●●●●●●●●●●●●●●●●●]
```

- **请求级并行**：不同请求共享 GPU 计算资源
- **解码时间填充**：短回复完成后立即处理新请求
- **吞吐量大幅提升**：实测可提升 10-50 倍吞吐量

### 2.2 TensorRT-LLM

**TensorRT-LLM** 是 NVIDIA 官方推出的大模型推理优化框架，基于 TensorRT 深度学习推理引擎，为 NVIDIA GPU 提供极致推理性能。

#### TensorRT-LLM 架构

```
┌─────────────────────────────────────────────┐
│              TensorRT-LLM API               │
│  （Python / C++ 接口，模型定义与配置）          │
├─────────────────────────────────────────────┤
│            TensorRT Engine                   │
│  （优化后的推理引擎：Kernel Fusion、         │
│   图优化、INT8/FP8 量化）                    │
├─────────────────────────────────────────────┤
│              CUDA / cuBLAS / cuDNN           │
│         （NVIDIA 官方 GPU 加速库）            │
├─────────────────────────────────────────────┤
│           NVIDIA GPU (A100/H100/L40S)        │
└─────────────────────────────────────────────┘
```

#### 核心优化技术

| 优化技术 | 说明 | 效果 |
|---------|------|------|
| **Kernel Fusion** | 将多个相邻 CUDA Kernel 融合为一个，减少显存访问 | 提升 20-50% 性能 |
| **Flash Attention** | IO-aware 的精确注意力算法，降低 HBM 访问 | 提升 3-5× 注意力速度 |
| **FP8 量化（Hopper）** | H100 支持的 8-bit 浮点格式 | 2× 吞吐提升 |
| **In-Flight Batching** | 类似 Continuous Batching，请求级并行 | 提升资源利用率 |
| **Weight-Only Quantization** | 仅量化权重，激活值保持高精度 | 精度损失极小 |
| **KV Cache INT8** | KV Cache 使用 INT8 存储 | 减少 50% KV Cache 显存 |

#### TensorRT-LLM vs vLLM

| 维度 | TensorRT-LLM | vLLM |
|------|-------------|------|
| **性能** | 极高（NVIDIA 深度优化） | 高（PagedAttention 优化） |
| **硬件** | NVIDIA GPU（Ampere/Hopper） | 各类 GPU |

async def chat_completions(request: ChatCompletionRequest):
    """
    聊天补全接口（同步版本）

    请求示例：
    {
        "messages": [
            {"role": "system", "content": "你是机器人助手"},

            max_tokens=request.max_tokens,

        )

        # Step 3: 调用 vLLM 推理（同步调用，异步用 add_request）
        outputs = llm_engine.generate([prompt], sampling_params)
        output = outputs[0]

        # Step 4: 构造响应
        response = {
            "id": f"chatcmpl-{hash(prompt) % 1000000}",
            "model": MODEL_NAME,
            "choices": [
                {
                    "index": 0,
                    "message": {
                        "role": "assistant",
                        "content": output.outputs[0].text
                    },
                    "finish_reason": "stop"
                }
            ],
            "usage": {
                "prompt_tokens": output.prompt_token_ids.__len__(),
                "completion_tokens": len(output.outputs[0].token_ids),
                "total_tokens": output.prompt_token_ids.__len__() + len(output.outputs[0].token_ids)
            }
        }
        return response

    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))


@app.post("/v1/chat/completions/stream")
async def chat_completions_stream(request: ChatCompletionRequest):
    """
    聊天补全接口（流式版本）

    使用 Server-Sent Events (SSE) 逐Token输出
    """
    if not request.stream:
        raise HTTPException(
            status_code=400,
            detail="请使用 /v1/chat/completions 接口进行非流式调用"
        )

    async def generate_stream():
        try:
            # 将消息转换为 Prompt
            prompt = messages_to_prompt(request.messages)

            # 配置采样参数
            sampling_params = SamplingParams(
                temperature=request.temperature,
                top_p=request.top_p,
                max_tokens=request.max_tokens,
                stop=request.stop,
            )

            # 创建异步请求
            result_generator = llm_engine.generate(
                [prompt],
                sampling_params,
                sampling_type=vllm.SamplingType.DOUBLE_STOCHASTIC  # 流式采样
            )

            # 逐 Token 流式输出
            async for output in async_stream_output(result_generator):
                chunk = {
                    "id": f"chatcmpl-stream",
                    "choices": [{
                        "delta": {"content": output},
                        "index": 0
                    }]
                }
                yield f"data: {chunk}\n\n"

            yield "data: [DONE]\n\n"

        except Exception as e:
            error_chunk = {"error": str(e)}
            yield f"data: {error_chunk}\n\n"

    return StreamingResponse(
        generate_stream(),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "Connection": "keep-alive"}
    )


# ============================================================
# 辅助函数
# ============================================================

def messages_to_prompt(messages: List[ChatMessage]) -> str:
    """
    将消息列表转换为 LLM 输入 Prompt
    兼容 Llama-2-chat 格式
    """
    prompt = ""
    for msg in messages:
        if msg.role == "system":
            prompt += f"[INST] <<SYS>>\n{msg.content}\n<</SYS>>\n[/INST]\n"
        elif msg.role == "user":
            prompt += f"[INST] {msg.content} [/INST]\n"
        elif msg.role == "assistant":
            prompt += f"{msg.content}\n"
    return prompt


async def async_stream_output(result_generator):
    """
    将 vLLM 的同步生成器转换为异步生成器（逐 Token 输出）
    """
    for output in result_generator:
        # 输出.outputs[0].text_delta 是当前新增的文本
        yield output.outputs[0].text
        # 让出控制权，允许其他协程执行
        await asyncio.sleep(0)

### 3.3 并发与限流

生产环境中，API 服务需要面对多用户并发访问，必须设计合理的限流策略保护模型服务不被压垮。

#### 限流策略

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| **固定窗口（Fixed Window）** | 每时间窗口允许 N 次请求 | 简单场景 |
| **滑动窗口（Sliding Window）** | 时间窗口平滑滑动，更精确 | 通用 API |
| **令牌桶（Token Bucket）** | 以恒定速率补充令牌，支持突发流量 | 最佳实践 |
| **漏桶（Leaky Bucket）** | 以恒定速率处理请求，平滑输出 | 流量整形 |

**令牌桶算法**：

$$
\text{可用令牌数} = \min\left(\text{桶容量}, \text{当前令牌数} + \text{补充速率} \times \Delta t\right)
$$

请求消耗 1 个令牌，令牌不足则拒绝。

```python
# 基于令牌桶的限流实现
import time
import asyncio
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class TokenBucket:
    """
    令牌桶限流器

    参数:
        rate: 令牌补充速率（令牌/秒）
        capacity: 桶容量（最大令牌数）
    """
    rate: float
    capacity: float
    current_tokens: float = field(init=False)
    last_update: float = field(init=False)

    def __post_init__(self):
        self.current_tokens = self.capacity
        self.last_update = time.monotonic()

    def _refill(self):
        """补充令牌（基于时间流逝）"""
        now = time.monotonic()
        elapsed = now - self.last_update
        # 补充令牌
        self.current_tokens = min(
            self.capacity,
            self.current_tokens + elapsed * self.rate
        )
        self.last_update = now

    def consume(self, tokens: int = 1) -> bool:
        """
        尝试消耗令牌

        返回:
            True: 成功获取令牌，请求可以通过
            False: 令牌不足，请求被拒绝
        """
        self._refill()
        if self.current_tokens >= tokens:
            self.current_tokens -= tokens
            return True
        return False

    def wait_time(self) -> float:
        """获取还需等待多少秒才能获得令牌"""
        self._refill()
        if self.current_tokens >= 1:
            return 0.0
        return (1 - self.current_tokens) / self.rate


# 全局限流器实例（每个用户/每个模型一个）
user_rate_limiters: dict[str, TokenBucket] = {}
model_rate_limiters: dict[str, TokenBucket] = {}

# 默认配置：每个用户每分钟 60 次调用，每个模型每分钟 120 次调用
DEFAULT_USER_RATE = 60 / 60    # 60 次/分钟
DEFAULT_MODEL_RATE = 120 / 60  # 120 次/分钟


def get_rate_limiter(limiter_id: str, rate: float, capacity: float) -> TokenBucket:
    """获取或创建限流器"""
    if limiter_id not in user_rate_limiters:
        user_rate_limiters[limiter_id] = TokenBucket(rate=rate, capacity=capacity)
    return user_rate_limiters[limiter_id]


# ============================================================
# FastAPI 限流中间件
# ============================================================
from fastapi import Request, HTTPException
from fastapi.responses import JSONResponse

@app.middleware("http")
async def rate_limit_middleware(request: Request, call_next):
    """请求限流中间件"""

    # 跳过健康检查
    if request.url.path == "/health":
        return await call_next(request)

    # 获取限流标识（按用户 ID 或 IP）
    client_id = request.headers.get("X-User-ID", request.client.host)
    model_name = request.state.model if hasattr(request.state, 'model') else "default"

    # 用户级限流：每分钟 60 次
    user_limiter = get_rate_limiter(
        f"user:{client_id}",
        rate=DEFAULT_USER_RATE,
        capacity=DEFAULT_USER_RATE * 2  # 允许 2 分钟 burst
    )

    if not user_limiter.consume():
        wait = user_limiter.wait_time()
        return JSONResponse(
            status_code=429,
            content={
                "error": "rate_limit_exceeded",
                "message": f"请求过于频繁，请等待 {wait:.1f} 秒后重试",
                "retry_after": int(wait) + 1
            },
            headers={"Retry-After": str(int(wait) + 1)}
        )

    response = await call_next(request)
    return response
```

\text{最大并发数} = \left\lfloor \frac{\text{GPU 显存}}{\text{单请求显存占用}} \right\rfloor
$$

ROS2 大模型服务客户端节点

from rclpy.action import ActionClient

        """
        instruction = msg.data
        self.get_logger().info(f"收到指令: {instruction}")

        # 在线程池中执行 HTTP 请求（不阻塞 ROS2 主循环）
        future = self.executor.submit(self._call_llm_sync, instruction)

    # ============================================================
    # 大模型服务调用
    # ============================================================

    def _call_llm_sync(self, instruction: str) -> Optional[dict]:
        """
        同步调用大模型服务

        参数:
            instruction: 自然语言指令

        返回:
            模型响应字典，失败返回 None
        """
        try:
            endpoint = f"{self.llm_url}/v1/chat/completions"
            headers = {"Content-Type": "application/json"}

            payload = {
                "messages": [
                    {
                        "role": "system",
                        "content": """你是一个具身智能机器人助手。请根据用户指令，
                        给出机器人应该执行的动作序列。用简洁的步骤描述。"""
                    },
                    {
                        "role": "user",
                        "content": instruction
                    }
                ],
                "temperature": self.temperature,
                "max_tokens": self.max_tokens,
                "stream": False
            }

            # 发送 HTTP POST 请求
            response = requests.post(
                endpoint,
                json=payload,
                headers=headers,
                timeout=self.timeout
            )

            if response.status_code == 200:
                return response.json()
            else:
                self.get_logger().error(
                    f"LLM 服务返回错误: {response.status_code} - {response.text}"
                )
                return None

        except requests.exceptions.Timeout:
            self.get_logger().error("LLM 服务调用超时")
            return None
        except requests.exceptions.ConnectionError:
            self.get_logger().error(f"无法连接到 LLM 服务: {self.llm_url}")
            return None
        except Exception as e:
            self.get_logger().error(f"LLM 服务调用异常: {str(e)}")

        处理大模型响应

        try:
            result = future.result()
            if result is None:
                return

            # 提取模型回复
            choices = result.get("choices", [])
            if choices:
                content = choices[0].get("message", {}).get("content", "")
                result_msg = String()
                result_msg.data = content
                self.result_publisher.publish(result_msg)
                self.get_logger().info(f"LLM 响应: {content[:100]}...")

        except Exception as e:
            self.get_logger().error(f"处理 LLM 响应异常: {str(e)}")

    # ============================================================
    # 异步调用版本（推荐使用）
    # ============================================================

    async def call_llm_async(self, instruction: str) -> Optional[dict]:

        import aiohttp

        async with aiohttp.ClientSession() as session:

            payload = {
                "messages": [
                    {"role": "system", "content": "你是机器人助手。"},
                    {"role": "user", "content": instruction}
                ],
                "temperature": self.temperature,
                "max_tokens": self.max_tokens,
                "stream": False
            }

            try:
                async with session.post(endpoint, json=payload, timeout=self.timeout) as resp:
                    if resp.status == 200:
                        return await resp.json()
                    else:
                        self.get_logger().error(f"服务错误: {resp.status}")
                        return None
            except asyncio.TimeoutError:
                self.get_logger().error("异步调用超时")
                return None

    def destroy_node(self):
        """清理资源"""
        self.executor.shutdown(wait=True)
        super().destroy_node()


# ============================================================
# 主程序入口
# ============================================================

def main(args=None):

if __name__ == "__main__":

### 4.3 机器人实时推理优化

机器人实时推理面临的核心挑战是**延迟确定性**：感知→决策→控制的全链路延迟必须稳定在可接受范围内（通常 < 100ms），否则机器人反应迟钝甚至失控。

#### 全链路延迟构成

$$
T_{\text{total}} = T_{\text{感知}} + T_{\text{通信}} + T_{\text{推理}} + T_{\text{决策}} + T_{\text{控制}}
$$

其中 $T_{\text{推理}}$ 由大模型决定，是最大的不确定性来源。

#### 优化策略

**策略 1：预测性 Prefill（Pre-computation）**

当机器人知道即将进入某场景时，提前触发 Prefill 阶段，将 Prompt 的 KV Cache 准备好：

```python
class PredictivePrefill:
    """
    预测性 Prefill：提前计算常用场景的 KV Cache

    例如：当机器人进入厨房区域时，
    预先准备好"厨房操作"相关的 system prompt 的 KV Cache
    """

    def __init__(self, llm_engine):
        self.llm = llm_engine
        self.cache_store = {}  # 场景名 -> KV Cache

    def register_scene(self, scene_name: str, system_prompt: str):
        """
        预注册一个场景的 system prompt

        参数:
            scene_name:  场景名称，如 "kitchen"
            system_prompt: 该场景的系统提示词
        """
        # 使用 vLLM 的提前编码功能
        # 注意：不同模型的 prefix 不可混用
        self.cache_store[scene_name] = {
            "prompt": system_prompt,
            # "kv_cache": 预计算的 cache 对象
        }
        print(f"场景 '{scene_name}' 已注册")

    def get_scene_prompt(self, scene_name: str, user_input: str) -> str:
        """
        获取该场景的完整 prompt
        实际使用时应利用预计算的 KV Cache 跳过 Prefill
        """
        if scene_name in self.cache_store:
            return self.cache_store[scene_name]["prompt"] + "\n" + user_input
        return user_input
```

**策略 2：投机解码（Speculative Decoding）**

使用小型"draft"模型快速生成候选 token序列，再用大型"target"模型验证，并行处理以加速：

$$
\text{加速比} \approx \frac{1}{\alpha \cdot (1 - \gamma) + \beta \cdot \gamma}
$$

其中 $\alpha$ 是 draft 模型延迟比例，$\beta$ 是验证开销，$\gamma$ 是接受率。

```python
# 投机解码示意（概念代码）
async def speculative_decode(
    target_model,
    draft_model,
    prompt: str,
    target_sampling_params,
    draft_sampling_params,
    gamma: int = 4
):
    """
    投机解码：draft 模型生成 gamma 个 token，target 模型并行验证

    参数:
        target_model:  大模型（如 Llama-70B）
        draft_model:   小模型（如 Llama-7B）
        prompt:        输入 prompt
        gamma:         每轮 draft token 数量
    """
    # Step 1: Prefill prompt
    target_outputs = target_model.generate([prompt], target_sampling_params)

    # Step 2: 循环验证 draft tokens
    while not target_outputs[0].outputs[0].finish_reason:
        # Draft 模型生成 gamma 个候选 token
        draft_outputs = draft_model.generate(
            [prompt + target_outputs[0].outputs[0].text],
            draft_sampling_params
        )
        draft_tokens = draft_outputs[0].outputs[0].token_ids[-gamma:]

        # Target 模型对这些 token 进行"验证"（一次前向传播）
        # 实际上是一次性计算所有 draft tokens 的 logits
        target_logits = target_model.compute_logits(
            prompt_tokens + draft_tokens,
            target_sampling_params
        )

        # 逐个 token 验证，接受的 token 加入输出
        accepted_tokens = []
        for i, draft_tok in enumerate(draft_tokens):
            target_prob = torch.softmax(target_logits[i], dim=-1)[draft_tok]
            # 如果 target 模型的概率较高，则接受
            if target_prob > 0.5:  # 可调整阈值
                accepted_tokens.append(draft_tok)
            else:
                # target 重新采样
                accepted_tokens.append(torch.multinomial(torch.softmax(target_logits[i], dim=-1), 1))
                break  # 拒绝后停止

        # 更新输出
        prompt += "".join([draft_model.tokenizer.decode([t]) for t in accepted_tokens])

    return prompt
```

**策略 3：连续推理（Continuous Inference）**

保持模型始终处于"热"状态，避免冷启动延迟：

- 服务启动后立即执行一次 dummy 推理预热
- 维护常驻内存池，避免显存碎片化
- 使用 `keep alive` 机制维持连接

```python
# 服务启动后预热
@app.on_event("startup")
async def warm_up():

        dummy_params = SamplingParams(
            temperature=0.0,

        )
        llm_engine.generate(["Hello"], warm_up_params)
        await asyncio.sleep(0.5)
    logger.info("预热完成！")
```

# 使用 FastAPI + Uvicorn 启动服务（推荐）

def start_with_fastapi(args):
    """
    使用 FastAPI + Uvicorn 启动 vLLM 服务
    提供完整的 REST API（与 OpenAI API 兼容）
    """
    import uvicorn
    from fastapi import FastAPI

    # 设置环境变量
    os.environ["VLLM_MODEL"] = args.model
    os.environ["VLLM_PORT"] = str(args.port)

    # 动态配置 vLLM（通过 FastAPI lifespan 加载模型）
    from contextlib import asynccontextmanager

    llm_instance = None

    @asynccontextmanager
    async def lifespan(app: FastAPI):
        global llm_instance
        print(f"正在加载模型: {args.model} ...")
        llm_instance = LLM(
            model=args.model,
            tensor_parallel_size=args.tensor_parallel_size,
            gpu_memory_utilization=args.gpu_memory_utilization,
            max_model_len=args.max_model_len,
            dtype=args.dtype,
            quantization=args.quantization,
            trust_remote_code=True,
        )
        # 将实例存入 app state 供后续使用
        app.state.llm = llm_instance
        print("模型加载完成！")
        yield
        print("服务关闭中...")

    app = FastAPI(title="vLLM 推理服务", lifespan=lifespan)

    # 健康检查端点
    @app.get("/health")
    async def health():
        return {"status": "healthy", "model": args.model}

    # 启动 uvicorn
    print(f"启动服务: http://{args.host}:{args.port}")
    uvicorn.run(
        app,
        host=args.host,
        port=args.port,
        log_level="info"
    )
