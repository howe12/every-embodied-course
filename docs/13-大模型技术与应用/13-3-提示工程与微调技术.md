## 1. 提示工程基础

### 1.1 什么是提示工程

**提示工程（Prompt Engineering）** 是研究和优化提示词（Prompt）设计的一门学科，旨在通过精心设计的输入文本来更有效地引导大语言模型（LLM）完成各种任务。与传统机器学习需要重新训练模型不同，提示工程不需要修改任何模型参数，仅通过优化输入文本就能显著提升模型输出质量。

提示工程的核心价值在于：

| 优势 | 说明 |
|------|------|
| **零成本迭代** | 无需训练或微调模型，仅修改文本即可 |
| **快速验证** | 可即时测试和优化，缩短开发周期 |
| **灵活适配** | 同一个模型通过不同Prompt可完成多种任务 |
| **能力挖掘** | 激发模型已学到的但未被充分利用的潜在能力 |

### 1.2 提示词的四大构成要素

一个完整的提示词通常由以下四个核心要素组成：

**（1）指令（Instruction）**

明确告诉模型要完成什么任务，是提示词中最核心的部分。

```python
# 示例
instruction = "请将以下中文句子翻译成英文"
```

**（2）上下文（Context）**

为模型提供必要的背景信息，帮助模型理解任务场景和约束条件。

```python
# 示例
context = "你是一位拥有10年经验的资深软件架构师，正在为一家互联网金融公司设计系统架构"
```

**（3）输入数据（Input）**

需要模型处理的具体内容，可以是文本、问题、数据等。

```python
# 示例
input_data = "设计一个高并发的用户积分系统，需要支持每秒10万次积分查询和1万次积分更新"
```

**（4）输出格式（Output Format）**

明确期望的输出形式，如JSON、Markdown、列表等。

```python
# 示例
output_format = "请以JSON格式返回，字段包括：system_design（系统设计说明）、tech_stack（技术栈列表）、database_schema（数据库表结构）"
```

**四要素整合示例**：

```python
# 完整提示词示例
prompt = """
# 指令
你是一位资深技术面试官，请对候选人的代码进行Code Review。

# 上下文
候选人应聘职位是高级Python后端工程师，有5年开发经验。这是他实现的一个用户认证模块。

# 输入数据
```python
def check_user(username, password):
    users = db.query("SELECT * FROM users")
    for user in users:
        if user.username == username and user.password == password:
            return True
    return False
```

# 输出格式
请以Markdown格式返回，包含：问题列表（每个问题标出严重程度）、修改建议、优化后的代码。
"""
```

### 1.3 Zero-shot 提示

**Zero-shot（零样本）提示** 是指不给模型提供任何示例，直接通过自然语言指令让模型完成任务。这是大模型最基本的使用方式，利用模型在预训练阶段学到的通用能力。

```python
# Zero-shot 示例
zero_shot_prompt = """
请判断以下评论的情感是正面、负面还是中性：

评论："这家餐厅的菜品非常一般，环境也比较嘈杂。"
"""

# Zero-shot 多任务示例
zero_shot_multi = """
请完成以下三个任务：
1. 将"Hello, world!"翻译成中文
2. 计算：256 × 17 = ?
3. 总结：人工智能的三大核心技术
"""
```

**Zero-shot的适用场景**：

| 场景 | 示例 | 效果 |
|------|------|------|
| 简单明确的任务 | 翻译、摘要、分类 | ✅ 效果好 |
| 通用知识问答 | 解释概念、回答事实问题 | ✅ 效果好 |
| 复杂推理任务 | 多步数学题、逻辑推理 | ⚠️ 需要CoT辅助 |
| 高度专业化任务 | 领域术语、专业流程 | ❌ 效果不稳定 |

### 1.4 Few-shot 提示

**Few-shot（少样本）提示** 是指在提示词中提供少量（通常1~5个）输入-输出示例，帮助模型理解任务模式。Few-shot本质上是**隐式的上下文学习（In-Context Learning）**，通过示例激活模型与当前任务相关的知识子集。

```python
# Few-shot 示例：情感分类
few_shot_prompt = """
请判断评论的情感是正向还是负向：

示例1：
评论："味道很棒，下次还会再来！"
情感：正向

示例2：
评论："等了一个小时还没上菜，太失望了。"
情感：负向

示例3：
评论："环境不错，但价格偏贵。"
情感：中性

现在请判断：
评论："服务态度很差，菜品也不新鲜。"
情感：
"""
```

**Zero-shot vs Few-shot 对比**：

| 对比维度 | Zero-shot | Few-shot |
|----------|-----------|----------|
| 示例数量 | 0个 | 1~5个 |
| 适用场景 | 简单通用任务 | 复杂或特殊格式任务 |
| 效果稳定性 | 相对较低 | 更稳定可靠 |
| Token消耗 | 较少 | 较多 |

**Few-shot设计技巧**：

| 技巧 | 说明 |
|------|------|
| **示例多样性** | 覆盖不同类型的输入，包括边界情况 |
| **标签一致性** | 输出标签格式统一，避免混淆 |
| **数量控制** | 通常2~5个最佳，过多反而分散注意力 |
| **先简单后复杂** | 按难度递增排列示例顺序 |

### 1.5 思维链提示（Chain-of-Thought, CoT）

**思维链（CoT）** 是由Google研究人员在2022年提出的提示技术，核心思想是引导模型在给出最终答案之前，先展示完整的推理过程。

**为什么CoT有效？**

1. **过程监督**：将最终答案的正确性转化为推理步骤的正确性
2. **知识激活**：强制模型调用与问题相关的中间知识
3. **错误定位**：推理过程便于发现和修正错误

**标准CoT格式（Few-shot CoT）**：

```python
# Few-shot CoT 示例
cot_prompt = """
请在回答数学问题时，展示完整的推理过程。

示例1：
问题：小明有12个苹果，给了小红5个，又从小刚那里得到3个，现在有多少个？
推理：
1. 小明开始有12个苹果
2. 给小红5个：12 - 5 = 7个
3. 从小刚得到3个：7 + 3 = 10个
答案：10个

示例2：
问题：一件商品原价比进价多80元，售价是150元，利润是多少元？
推理：
1. 利润 = 售价 - 进价
2. 已知：售价 = 150元，原价（售价）比进价多80元
3. 因此：进价 = 150 - 80 = 70元
4. 利润 = 150 - 70 = 80元
答案：80元

现在请回答：
问题：一辆汽车以60km/h的速度行驶3小时，然后以80km/h的速度行驶2小时，总行程是多少公里？
"""
```

**Zero-shot CoT**：

不需要手工设计示例，只需在问题后添加触发词 `"让我们一步步思考"` 即可诱导模型展示推理过程。

```python
# Zero-shot CoT
zeroshot_cot_prompt = """
一家书店有120本书，第一天卖了30本，第二天卖了剩下的1/3，
还剩多少本书？

让我们一步步思考：
"""
```

**进阶CoT触发词**：

| 触发词（中文） | 触发词（英文） | 适用场景 |
|----------------|----------------|----------|
| 让我们一步步思考 | Let's think step by step | 通用推理任务 |
| 仔细分析这个问题 | Let me analyze this carefully | 分析类问题 |
| 首先...然后...最后 | First...Then...Finally | 流程规划问题 |
| 让我们逐步推理 | Let's reason step by step | 数学逻辑问题 |

### 1.6 结构化输出

大模型可以输出多种结构化格式，便于程序解析或直接使用。

**JSON格式输出**：

json_output_prompt = """

## 2. 高级提示技术

### 2.1 角色扮演提示（Role Prompting）

通过设定特定的角色身份，可以让模型以该角色的视角和专业背景来回答问题，这往往能显著提升回答的质量和专业性。

**基础角色设定**：

```python
role_prompt = """
你是一位经验丰富的ROS2机器人软件架构师，具有8年机器人开发经验，精通ROS2、导航算法和运动控制。

请解释ROS2中话题（Topic）和服务（Service）的区别，并说明各自适用的场景。
"""
```

**带约束的角色设定**：

```python
role_constrained_prompt = """
你是一位严谨的大学计算机教授，在授课时注重理论与实践结合，善于用生活中的例子解释抽象概念。

请讲解"设计模式"中的单例模式（Singleton Pattern）：
- 用通俗易懂的语言解释概念（100字以内）
- 提供一个Python代码实现示例
- 列出2个常见的使用场景
- 指出使用单例模式需要注意的问题
"""
```

**角色设定的核心技巧**：

| 技巧 | 说明 | 示例 |
|------|------|------|
| 明确专业背景 | 指定相关领域和工作年限 | "10年经验的自动驾驶算法工程师" |
| 设定回答风格 | 指定语气、详略程度 | "简洁明了，使用专业术语" |
| 提供受众信息 | 指定目标读者水平 | "面向初学者，用类比解释概念" |
| 添加行为约束 | 明确应该做什么、不做什么 | "只提供代码思路，不给完整实现" |

### 2.2 思维树（Tree of Thoughts, ToT）

**思维树（ToT）** 是对CoT的扩展，它不是线性推理，而是探索多条可能的思考路径，在每条路径上继续分支，最终选择最优路径。ToT特别适合需要多步规划或探索多种解决方案的复杂任务。

**ToT的核心步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 思维分解 | 将问题分解为若干思考步骤 |
| 2 | 候选生成 | 每个步骤生成多个候选想法 |
| 3 | 状态评估 | 评估每个状态是否可能达到目标 |
| 4 | 搜索回溯 | 选择最佳路径，必要时回溯 |

**ToT vs CoT 对比**：

```
CoT（线性推理）：
A → B → C → D（一个答案，不支持回溯）

ToT（树状探索）：
         A（问题）
      ↙    ↓    ↘
    B1     B2     B3（三个初始方案）
   ↙↓    ↙↓     ↙↓
  C1 C2  C3 C4   C5 C6（分支扩展）
      ↓       ↓
  评估选择最优路径
```

**ToT实现示例**：

```python
tot_prompt = """
你是一个战略规划专家，需要为机器人创业公司制定产品路线图。

## 第一步：问题分解
请列出制定产品路线图需要考虑的核心维度（例如：技术可行性、市场需求、竞争分析、资源限制等）。

## 第二步：每个维度生成多个方案
对每个维度，生成2~3个不同方向的方案，并简要说明优劣。

## 第三步：评估组合
从所有方案中，选择最合理的组合，给出最终建议。

## 第四步：制定路线图
基于选择的方案，制定6个月的产品路线图（按月列出里程碑）。

请确保每个步骤都有清晰的推理过程，并在选择方案时说明理由。
"""
```

### 2.3 自我一致性（Self-Consistency）

**自我一致性（Self-Consistency）** 是由Google研究人员提出的CoT进阶技术。核心思想是：对同一个问题，使用多个不同的推理路径生成多个答案，然后通过投票选择最一致的答案作为最终输出。

**为什么自我一致性有效？**

人类的推理过程往往存在随机性，大模型也类似。同一推理路径在多次采样中可能产生波动，而正确答案在所有合理推理路径中出现的概率更高。通过多数投票，可以过滤掉推理过程中的噪声和错误。

**自我一致性算法流程**：

```python
# 自我一致性实现伪代码
def self_consistency(prompt, model, n_samples=5):
    """
    prompt: 原始提示词（通常包含CoT触发词）
    model: 大模型接口
    n_samples: 采样次数
    """
    # 步骤1：生成多个不同的推理路径和答案
    answers = []
    for i in range(n_samples):
        # 每次采样使用不同的temperature，增加随机性
        response = model.generate(
            prompt,
            temperature=0.7,  # 较高温度增加多样性
            do_sample=True
        )
        # 从response中提取最终答案
        answer = extract_answer(response)
        answers.append(answer)
    
    # 步骤2：多数投票选择最一致的答案
    final_answer = majority_vote(answers)
    return final_answer
```

**自我一致性示例**：

```python
self_consistency_prompt = """
问题：一个水池有进水管和出水管。进水管8小时可以注满，出水管6小时可以放完。如果两管同时打开，需要多少小时才能注满水池？

请一步步推理，然后给出最终答案。
（请用"答案：X小时"的格式结尾）
"""

# 使用自我一致性进行5次采样
# 假设5次采样得到了以下答案：[12, 12, 10, 12, 12]
# 则最终答案：12小时（出现3次，多数票）
```

**Self-Consistency vs Standard CoT**：

| 对比维度 | Standard CoT | Self-Consistency |
|----------|--------------|------------------|
| 推理路径 | 1条 | 多条（通常5~10条）|
| 计算成本 | 低 | 高（需多次采样）|
| 准确率提升 | 基准 | 显著提升（尤其数学题）|
| 适用场景 | 简单推理 | 复杂推理、事实性问题 |

### 2.4 自动提示工程（AutoPrompt / APE）

手工设计Prompt需要大量试错，成本较高。**自动提示工程（Automatic Prompt Engineering, APE）** 旨在通过算法自动搜索和优化Prompt。

**APE的核心思想**：

```python
# APE伪代码
def automatic_prompt_engineering(task_description, train_set, n_iterations=10):
    """
    自动提示词优化流程
    """
    # 步骤1：生成候选Prompt集合
    # 基于LLM生成多个不同风格的候选Prompt
    candidate_prompts = llm.generate(
        f"针对以下任务，生成5个不同风格的提示词：{task_description}"
    )
    
    # 步骤2：评估每个候选Prompt的效果
    prompt_scores = {}
    for prompt in candidate_prompts:
        score = evaluate_prompt(prompt, train_set)  # 在验证集上评估
        prompt_scores[prompt] = score
    
    # 步骤3：选择最优Prompt，并进行迭代优化
    best_prompt = max(prompt_scores, key=prompt_scores.get)
    
    for _ in range(n_iterations):
        # 步骤4：基于最优Prompt进行局部搜索（改写、添加约束等）
        mutated_prompts = mutate_prompts(best_prompt)
        # 重复评估过程...
    
    return best_prompt
```

**常见的自动提示工程技术**：

| 方法 | 提出者 | 核心思想 |
|------|--------|----------|
| **AutoPrompt** | Shin et al., 2020 | 用梯度搜索自动生成触发词（Trigger Tokens） |
| **APE** | Zhou et al., 2022 | LLM生成候选Prompt，评估选择最优 |
| **Prompt o1** | OpenAI | 模型自己优化自己的Prompt |
| **DSPy** | Stanford | 用编译方法优化Prompt管道 |

## 3. 模型微调技术

### 3.1 为什么需要微调

大语言模型（LLM）在预训练阶段学习到了海量的通用知识——从互联网文本中掌握了语言规律、世界常识甚至编程能力。然而，这些通用能力并不能直接满足特定垂直场景的需求：

| 问题 | 示例 |
|------|------|
| **领域知识缺失** | 通用模型无法精准回答"本公司的报销流程是什么" |
| **风格不匹配** | 通用模型无法按照特定风格（如客服话术、技术文档）生成内容 |
| **术语偏差** | 通用模型在特定领域术语理解上存在偏差 |

**微调（Fine-tuning）** 的核心思想是：利用少量领域数据，在预训练模型的基础上继续训练，使模型适配特定任务或领域。

### 3.2 全量微调 vs 参数高效微调

| 对比维度 | 从头训练 | 全量微调 | 参数高效微调 |
|----------|----------|----------|--------------|
| 训练数据量 | 数十万亿Token | 数百万~数千万Token | 数千~数万Token |
| 计算成本 | 极高（数百万美元）| 高（数天~数周）| 低（几小时~几天）|
| 过拟合风险 | 低 | 高 | 低 |
| 领域适配性 | 差 | 好 | 好 |
| 训练时间 | 数月 | 数天~数周 | 数小时~数天 |
| 参数更新 | 全部 | 全部 | 仅少量参数 |

### 3.3 Full Fine-tuning（全参数微调）

**全参数微调** 是指在微调过程中更新模型的所有参数。模型在预训练权重的基础上，使用特定任务的数据继续训练，所有参数都会根据任务目标进行调整。

**全参数微调的特点**：

| 特点 | 说明 |
|------|------|
| 效果最好 | 理论上能达到最优任务性能 |
| 计算成本高 | 需要GPU显存与预训练相当 |
| 灾难性遗忘 | 可能丢失预训练学到的部分通用能力 |
| 部署困难 | 每个任务需要独立的大型模型 |

```python
# 全参数微调示例（使用Hugging Face Transformers）
from transformers import AutoModelForSequenceClassification, AutoTokenizer, TrainingArguments, Trainer

# 加载预训练模型
model_name = "bert-base-chinese"

### 3.4 LoRA（Low-Rank Adaptation）原理与实现

**LoRA**（Low-Rank Adaptation，低秩适配）是由Microsoft研究人员于2022年提出的参数高效微调方法。其核心思想是：预训练模型的全连接层权重矩阵通常是过参数化的（即秩较高），但实际任务只需要一个低秩的改变量。

**LoRA的核心原理**：

对于预训练模型的权重矩阵 $W_0 \in \mathbb{R}^{d \times k}$，LoRA不直接更新 $W_0$，而是添加一个低秩分解增量：

$$
W = W_0 + \Delta W = W_0 + BA
$$

其中：
- $B \in \mathbb{R}^{d \times r}$，$A \in \mathbb{R}^{r \times k}$
- $r \ll \min(d, k)$，称为**秩（Rank）**，是LoRA最核心的超参数
- 只有 $A$ 和 $B$ 是可训练的，$W_0$ 保持冻结

```
原始全参数更新（全量微调）：
    W = W₀ - η × ∇W Loss    （需要更新所有参数，显存占用大）

LoRA更新：
    W = W₀ + BA             （只更新A和B两个小矩阵，显存占用小）
    A ~ N(0, σ²),  B = 0    （初始化：A随机，B零初始化）
```

**LoRA的关键优势**：

| 优势 | 说明 |
|------|------|
| 参数效率高 | 参数量从 $d \times k$ 降低到 $r \times (d + k)$，通常 $r \ll \min(d,k)$ |
| 显存效率高 | 只训练LoRA参数，冻结预训练权重 |
| 训练稳定 | 无需调整大学习率，不易崩溃 |
| 可插拔 | 不同任务训练不同的LoRA权重，推理时可动态加载 |
| 避免遗忘 | 保留原始预训练能力 |

**LoRA超参数说明**：

| 超参数 | 说明 | 常用值 |
|--------|------|--------|
| `r`（Rank）| 低秩矩阵的秩，越大越接近全量微调 | 4, 8, 16, 32, 64 |
| `lora_alpha` | 缩放因子，通常设为 `r` 的2倍 | 16 |
| `lora_dropout` | Dropout概率 | 0.05, 0.1 |
| `target_modules` | 应用LoRA的模块 | `q_proj`, `v_proj`, `k_proj`, `o_proj` |

```python
# LoRA微调实战（使用PEFT库 + Hugging Face）

from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer

# 配置训练参数

    output_dir="./lora_output",

    model=model,

### 3.5 QLoRA（量化LoRA）

**QLoRA**（Quantized LoRA）是由University of Washington研究人员于2023年提出的方法，将量化技术与LoRA结合，可以在大幅降低显存占用的同时保持接近全量微调的效果。

**QLoRA的核心技术**：

| 技术 | 说明 |
|------|------|
| **NF4（4-bit NormalFloat）** | 针对神经网络权重分布优化设计的4位量化数据类型 |
| **双重量化（Double Quantization）** | 对量化常数本身再量化，进一步节省显存 |
| **分页优化器（Paged Optimizers）** | 将优化器状态分页到CPU内存，避免显存峰值 |

```python
# QLoRA微调实战（使用bitsandbytes + PEFT）

from peft import LoraConfig, get_peft_model, TaskType

# 4位量化配置（NF4）

    load_in_4bit=True,              # 启用4位量化加载
    bnb_4bit_quant_type="nf4",      # 使用NF4量化类型
    bnb_4bit_compute_dtype=torch.bfloat16,  # 计算时使用BF16
    bnb_4bit_use_double_quant=True, # 启用双重量化
)

# 加载预训练模型（QLoRA模式下可加载更大模型）
model_name = "tiiuae/falcon-rw-7b"  # 7B参数模型
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    quantization_config=bnb_config, # 应用量化配置
    device_map="auto",
)

# 配置LoRA（与普通LoRA相同）
lora_config = LoraConfig(
    r=64,
    lora_alpha=16,
    lora_dropout=0.05,
    task_type=TaskType.CAUSAL_LM,
    target_modules=["query_key_value"],  # Falcon模型特有的注意力模块名
)

model = get_peft_model(model, lora_config)

# 训练配置
training_args = TrainingArguments(
    output_dir="./qlora_output",
    num_train_epochs=3,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    learning_rate=2e-4,
    weight_decay=0.01,
    bf16=True,        # QLoRA使用BF16训练
    logging_steps=10,
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,
)

trainer.train()
```

### 3.6 Prefix Tuning / Adapter Tuning

**Prefix Tuning** 是另一种参数高效微调方法，其核心思想是在每个Transformer层的输入序列前面添加一组可学习的前缀向量，而不是修改模型权重。

```
标准Transformer层输入：
    [Input Tokens] → [Attention] → [FFN] → Output

Prefix Tuning输入：
    [Prefix] [Prefix] [Prefix] [Prefix] | [Input Tokens] → [Attention] → [FFN] → Output
     ↑可学习                          ↑冻结
```

```python
# Prefix Tuning示例（使用PEFT）
from peft import PromptTuningConfig, get_peft_model, TaskType

# Prefix Tuning配置
prefix_config = PromptTuningConfig(
    task_type=TaskType.CAUSAL_LM,
    num_virtual_tokens=20,    # 前缀token数量
    prompt_tuning_init="TEXT",
    prompt_tuning_init_text="以下是一个机器人控制任务：请逐步规划机器人的动作：",  # 初始化文本
)

model = get_peft_model(base_model, prefix_config)
```

**Adapter Tuning** 则是在Transformer层之间插入小型适配器模块：

```python
# Adapter核心结构伪代码
class AdapterLayer(nn.Module):
    def __init__(self, d_model, adapter_size=64):
        super().__init__()
        self.down_project = nn.Linear(d_model, adapter_size)  # 压缩
        self.up_project = nn.Linear(adapter_size, d_model)    # 恢复
        self.activation = nn.ReLU()
    
    def forward(self, x):
        # 学习残差适配，而非直接替换
        return x + self.up_project(self.activation(self.down_project(x)))
```

### 3.7 指令微调（Instruction Tuning）

**指令微调（Instruction Tuning）** 是一种特殊的微调范式，其核心是使用（指令, 输入, 输出）格式的数据对模型进行微调，使模型学会理解并遵循自然语言指令。

**指令微调与普通微调的区别**：

| 对比维度 | 普通微调 | 指令微调 |
|----------|----------|----------|
| 数据格式 | (输入, 输出) | (指令, 输入, 输出) |
| 数据内容 | 任务相关样本 | 多任务、多领域的指令样本 |
| 泛化能力 | 仅针对特定任务 | 泛化到未见过的指令 |
| 典型数据量 | 数千~数万 | 数千~数十万 |

**指令微调数据格式**：

```json
// 指令微调数据示例（Alpaca格式）
{
    "instruction": "将以下中文句子翻译成英文",
    "input": "今天天气真好，我们去公园散步吧。",
    "output": "The weather is really nice today. Let's take a walk in the park."
}
```

```python
# 指令微调训练流程
from transformers import AutoModelForCausalLM, AutoTokenizer

# 加载基础模型
model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b-hf")
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")

# 格式化训练数据（将instruction、input、output拼接为单一文本）
def format_instruction(example):
    """
    将数据格式化为模型输入格式
    常见格式：[INST] 指令 [/INST] 输入 [/INST] 输出
    """
    if example.get("input", ""):
        text = f"[INST] {example['instruction']} {example['input']} [/INST] {example['output']}"
    else:
        text = f"[INST] {example['instruction']} [/INST] {example['output']}"
    return {"text": text}

# 应用格式化
formatted_dataset = dataset.map(format_instruction)

# Tokenize
def tokenize_function(example):
    result = tokenizer(
        example["text"],
        truncation=True,
        max_length=512,
        padding="max_length",
    )
    # labels用于计算损失，只在output部分计算
    result["labels"] = result["input_ids"].copy()
    return result

tokenized_dataset = formatted_dataset.map(tokenize_function)
```

### 5.2 LoRA 微调实战（使用 PEFT 库）

```python

## 7. 参考答案

### 选择题答案

**1. 答案：C**

解析：Zero-shot（零样本）提示是指不给模型提供任何示例，直接通过自然语言指令完成任务；Few-shot（少样本）提示则需要提供1~5个输入-输出示例，帮助模型理解任务模式。两者最核心的区别就在于是否提供示例。

**2. 答案：B**

解析：Zero-shot CoT不需要手工设计包含推理过程的示例，只需在问题后添加触发词（如"让我们一步步思考"或"Let's think step by step"）即可诱导模型展示推理过程。Few-shot CoT才需要手工设计包含推理过程的示例。

**3. 答案：C**

解析：LoRA的核心原理是冻结原始预训练权重矩阵 $W_0$，通过低秩分解 $W = W_0 + BA$ 添加可训练的增量参数，其中 $A \in \mathbb{R}^{r \times k}$，$B \in \mathbb{R}^{d \times r}$，且 $r \ll \min(d, k)$。选项A描述的是全参数微调，选项B描述的是Adapter Tuning，选项D描述的是QLoRA的一部分技术。

### 简答题答案

**4. 答案：**

提示词的四大构成要素包括：

| 要素 | 作用 |
|------|------|
| **指令（Instruction）** | 明确告诉模型要完成什么任务，是提示词中最核心的部分，直接决定模型的任务方向 |
| **上下文（Context）** | 为模型提供必要的背景信息，帮助模型理解任务场景、角色设定和约束条件 |
| **输入数据（Input）** | 需要模型处理的具体内容，可以是文本、问题、代码或数据等 |
| **输出格式（Output Format）** | 明确期望的输出形式（如JSON、Markdown、列表等），使模型输出更易于程序解析或直接使用 |

**5. 答案：**

LoRA的工作原理：对于预训练模型的权重矩阵 $W_0 \in \mathbb{R}^{d \times k}$，LoRA不直接更新 $W_0$，而是添加一个低秩分解增量 $\Delta W = BA$，其中 $B \in \mathbb{R}^{d \times r}$，$A \in \mathbb{R}^{r \times k}$，且 $r \ll \min(d, k)$。最终模型权重为 $W = W_0 + BA$，只有 $A$ 和 $B$ 是可训练的，$W_0$ 保持冻结。

LoRA节省显存的原因：全参数微调需要为所有参数存储梯度（梯度大小与参数相同）和优化器状态（Adam优化器需要存储一阶和二阶动量，通常是参数量的2倍）。而LoRA只训练 $A$ 和 $B$ 两个小矩阵，参数量从 $d \times k$ 降低到 $r \times (d + k)$，因此梯度、优化器状态都大幅减少，从而节省显存。

以GPT-2中一个注意力权重矩阵为例：$d=768, k=768$，全参数微调需要更新的参数量为 $768 \times 768 = 589,824$；如果使用 $r=8$ 的LoRA，则参数量为 $8 \times (768 + 768) = 12,288$，仅为原来的约2%。

**6. 答案：**

指令微调与普通微调的区别：

| 对比维度 | 普通微调 | 指令微调 |
|----------|----------|----------|
| 数据格式 | (输入, 输出) | (指令, 输入, 输出) |
| 数据内容 | 任务相关样本 | 多任务、多领域的指令样本 |
| 泛化能力 | 仅针对特定任务 | 泛化到未见过的指令 |
| 典型数据量 | 数千~数万 | 数千~数十万 |

指令微调的核心数据格式（Alpaca格式）：
```json
{
    "instruction": "将以下中文句子翻译成英文",
    "input": "今天天气真好。",
    "output": "The weather is really nice today."
}
```

指令微调的目的是让模型学会理解并遵循自然语言指令，即使对于未见过的指令也能正确执行，具有良好的泛化能力。
